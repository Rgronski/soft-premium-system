# SPS Experience Blueprint

## Purpose

This document describes the intended workspace experience for SPS OS 1.0.

It is a user experience blueprint, not a technical specification.

It defines how the workspace should feel and guide the user during a normal working day inside Soft Premium System.

## 1. Cel SPS Workspace

SPS Workspace exists to help the user enter work quickly, understand current context immediately, and continue with confidence.

The workspace should reduce friction between intention and action.

The user should not need to reconstruct what happened before, what is active now, or what the next step should be.

## 2. Morning Experience

After opening SPS, the user should feel that the system is ready to continue the day, not asking them to start from zero.

The first experience should communicate:

* where the work currently stands,
* what context has been loaded,
* what deserves attention now,
* what the next recommended step is.

The morning experience should feel calm, clear, and directed.

It should help the user enter focus mode without searching through multiple places.

## 3. Workspace

The main workspace should act as a structured operating surface for work.

In SPS OS 1.0, the main screen should be understood as having these sections:

* current context,
* active workspace area,
* guidance area,
* key actions area,
* closing state visibility.

The workspace should show enough information to support action, but not overload the user with operational noise.

It should keep the user oriented in one place instead of forcing constant navigation.

## 4. The Conductor

In SPS OS 1.0, The Conductor has a narrow and practical role.

The Conductor:

* loads context,
* proposes the next step,
* guides the user.

The Conductor is not described here as an autonomous builder or decision-maker.

Its role in 1.0 is to help the user stay oriented and keep workflow continuity inside the workspace.

## 5. Główne akcje

SPS OS 1.0 exposes three primary actions:

* `▶ Zaczynamy`
* `💡 Mam pomysł`
* `🌙 Kończymy`

`▶ Zaczynamy` should help the user enter the current work context and begin the next meaningful step.

`💡 Mam pomysł` should let the user capture a new direction or idea without losing the current workspace flow.

`🌙 Kończymy` should support a clean end-of-day transition with visible closure.

## 6. Closing Experience

The end of the day should feel complete, not abrupt.

Closing experience should help the user understand:

* what was done,
* what remains active,
* what should happen next,
* whether the workspace is ready to resume later.

The user should leave SPS with confidence that the next session can start from a known state.

## 7. Design Goals

* Reduce startup friction at the beginning of the work session.
* Keep the user oriented in one main workspace.
* Present the next step clearly without overexplaining.
* Support calm, guided progress through the day.
* Make end-of-day closure explicit and understandable.
